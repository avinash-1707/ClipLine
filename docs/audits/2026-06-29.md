# Clipline Data-Flow Audit ‚Äî 2026-06-29

Whole-app, exhaustive data-flow audit. Scope: every entry-point flow across
`apps/api`, `apps/worker`, `apps/web`, and the shared `packages/timeline` /
`packages/jobs` contracts. No flows were capped or sampled.

**Context note.** Per `context/architecture.md` the system is single-user,
local, no auth, and trusts all local requests by design. Auth/access-control
gaps are therefore out of scope as defects, but a CORS exposure that widens
the trust boundary beyond "local requests" is flagged (F8) as informational.

## Findings

### F1 ‚Äî Asset deletion orphans live timeline references ‚Äî **High**

`deleteAsset` (`apps/api/src/services/assets.ts:60-82`) removes the Cloudinary
binaries and the `assets` row but never scrubs clips that reference that
`assetId` inside any project's `timeline` JSONB (nor `colorGrade.lutAssetId`).
There is no FK from the JSONB blob to `assets`, so nothing protects the
reference.

Consequences after a delete of an in-use asset:
- **Preview**: the clip silently renders black and produces no audio ‚Äî
  `engine.ts:189` (`if (!media) continue`) and `engine.ts:398-400`
  (`if (!video ‚Ä¶) return`) swallow the missing asset with no signal.
- **Export**: hard fails at `apps/api/src/routes/render.ts:53-54` with
  `asset <uuid> is not ready to render` (422). The message names a raw UUID;
  the user has no in-app way to find or remove the offending clip, so the core
  export flow is blocked with no recovery path.

**Fix.** On asset delete, run a transaction that either (a) refuses deletion
while the asset is referenced by any timeline, or (b) scrubs every matching
clip (and `lutAssetId`) from all project timelines before deleting the row.
Surface the affected project/clip either way.

---

### F2 ‚Äî SSE can miss the terminal event and hang forever ‚Äî **Medium**

`GET /render-jobs/:id/progress` (`apps/api/src/routes/render.ts:80-123`) reads
the job state with `await getRenderJob(id)` and *then*, inside the `streamSSE`
callback, subscribes with `renderEvents.on(id, onEvent)`. `renderEvents.emit`
is a synchronous `EventEmitter` fire (`apps/api/src/lib/queues.ts:108-127`).
If the worker completes in the await-gap between the read (status still
`rendering`) and the subscription, the terminal `completed`/`failed` event is
emitted with no listener and is lost. The client then receives only 15s
heartbeats (`render.ts:118`) and the export UI stays pinned at its last
percentage indefinitely.

**Fix.** Subscribe to `renderEvents` *before* reading the job row, or re-read
the job status immediately after subscribing and emit the terminal state if it
already finished (close the obvious race window).

---

### F3 ‚Äî Render status flips back to "rendering" (lost update) ‚Äî **Medium**

`render.on("progress")` and `render.on("completed")` are independent async
QueueEvents handlers (`apps/api/src/lib/queues.ts:102-122`). BullMQ delivers
events in order, but the handlers are not serialized: a slow
`updateRenderProgress` write can resolve *after* `markRenderCompleted`,
overwriting the row back to `status: "rendering", progress < 1`
(`services/render-jobs.ts:31-36` vs `38-49`). The render is done but
`GET /render-jobs/:id` permanently reports it as still rendering.

**Fix.** Make the progress write conditional ‚Äî e.g. `UPDATE ‚Ä¶ SET progress ‚Ä¶
WHERE id = $1 AND status = 'rendering'` ‚Äî so a terminal status can't be
clobbered, or drop progress writes once the terminal handler has run.

---

### F4 ‚Äî Non-atomic create+enqueue leaves permanently stuck rows ‚Äî **Medium**

Two flows insert a DB row and then enqueue a job in separate, non-atomic steps.
If the enqueue throws (`enqueueIngest`/`enqueueRender` raise `AppError(503)`
when Redis is down ‚Äî `apps/api/src/lib/queues.ts:48-63,34-46`):

- **Upload** (`routes/assets.ts:66-79`): the `assets` row is already inserted as
  `status: "processing"` and the Cloudinary original already uploaded. The row
  is never ingested, sits in `processing` forever, and the client polls it every
  2s indefinitely (`apps/web/lib/use-assets.ts:11`). The Cloudinary binary is
  orphaned.
- **Render** (`routes/render.ts:59-65`): the `render_jobs` row is inserted as
  `queued` and then stranded in `queued` forever.

**Fix.** Enqueue before the durable insert where possible, or roll back / mark
the row `failed` when the enqueue fails, and add a reconciler for rows that
never left their initial state.

---

### F5 ‚Äî Unbounded in-memory buffering of media ‚Äî **Medium**

Both the upload ingress and the worker download read entire media files into a
single `Buffer` with no size limit:

- API: `Buffer.from(await file.arrayBuffer())` ‚Äî `routes/assets.ts:59`, after
  `c.req.parseBody()` already materializes the multipart body.
- Worker: `Buffer.from(await response.arrayBuffer())` ‚Äî
  `apps/worker/src/lib/cloudinary.ts:57`.

A source clip up to the 120s export limit can be hundreds of MB to several GB.
Concurrent uploads or a large source can OOM the API or worker process.

**Fix.** Enforce a max upload size (reject early), stream the upload directly to
Cloudinary, and stream the worker download to disk
(`Readable.fromWeb(response.body).pipe(createWriteStream(dest))`) instead of
buffering.

### F6 ‚Äî Autosave issues concurrent PUTs; last-write-wins loses edits ‚Äî **Medium**

`useAutosave` (`apps/web/lib/use-autosave.ts:22-45`) debounces only the *timer*.
Once a save is in flight (awaiting `saveTimeline`), a new edit schedules a fresh
800ms timer that can fire and issue a *second* concurrent `PUT`. There is no
request sequencing or cancellation, and `saveTimeline` is a full-document
overwrite (`apps/api/src/services/projects.ts:43-50`), so if the two PUTs
complete out of order the older timeline silently overwrites the newer one. The
export pre-flight save (`export-dialog.tsx:49`) races autosave the same way.

Single-user, but reachable on a slow network with rapid edits.

**Fix.** Serialize saves (await the in-flight request before starting the next),
abort superseded requests, or add optimistic concurrency via `updatedAt`.

### F7 ‚Äî Timeline schema has no global size bounds ‚Äî **Low**

`timelineSchema` (`packages/timeline/src/schema.ts:317-323`) caps neither the
number of tracks nor total clips. `PUT timeline` only validates total duration
(`routes/projects.ts:61-67`). A timeline with thousands of tracks passes
validation, bloats the `projects.timeline` JSONB, and fans out render work.
Low given single-user.

**Fix.** Cap track count and total clip count in the schema.

---

### F8 ‚Äî CORS reflects any origin with no auth ‚Äî **Low (informational)**

`app.use(cors())` (`apps/api/src/index.ts:16`) uses the default permissive
policy. Combined with the intentional no-auth model, any website the user
visits can drive the local API in their browser ‚Äî create/delete projects and
assets, trigger renders (a CSRF / DNS-rebinding vector). This is distinct from
"auth" and widens the documented "trusts local requests" boundary to "trusts
any web page." Flagged per policy, not fixed.

**Fix (when revisited).** Restrict the allowed origin to the editor's own
localhost origin instead of reflecting `*`.

---

### F9 ‚Äî Transition duration unbounded vs. clip length ‚Äî **Low**

`transitionSchema.durationInFrames` (`packages/timeline/src/schema.ts:62-67`)
is any positive int. If it exceeds the adjacent clip's duration, the preview
progress `(intoWindow+1)/durationInFrames` (`lib/preview/resolve.ts:107`) never
reaches 1 within the clip, so the transition never visually completes. A
`transitionToNext` can also be set on a clip with no adjacent next clip
(dangling; ignored at `timeline-ops.ts:408`). Visual only.

**Fix.** Refine the schema/track validation to bound transition duration by the
shorter adjacent clip and require adjacency.

---

### F10 ‚Äî Asset kind trusts client MIME ‚Äî **Low**

`assetKindFromMime(file.type)` (`routes/assets.ts:19-23,54`) keys off the
client-supplied `file.type`. A mismatch is caught later by ffprobe
(`worker/.../ingest/index.ts:38`, `ffmpeg.ts:51`), so defense-in-depth exists,
but the original is uploaded to Cloudinary before any real content check.

**Fix.** Treat ingest probe as authoritative for `kind`; optionally sniff
content before the original upload.

---

### F11 ‚Äî Render does not scope assets to the project ‚Äî **Low**

`POST /render` resolves each clip's asset via `getAsset(assetId)`
(`routes/render.ts:52`) without checking `asset.projectId === projectId`. A
timeline could reference another project's asset and still render. Single-user,
so low impact; an integrity smell.

**Fix.** Scope the asset lookup to the project, or validate ownership.

---

### F12 ‚Äî Split replays entrance animation on the right half ‚Äî **Low**

`splitClip` (`apps/web/lib/timeline-ops.ts:262-267`) copies `animation` to the
right half, whose `localFrame` resets at the new `startFrame`, so a text/graphic
entrance animation re-plays at the split point. Minor visual.

**Fix.** Set the right half's animation preset to `none` (or carry forward
elapsed local frames).

---

## Verdict

The codebase is small, well-factored, and validates inputs consistently at the
zod boundary; the timeline-ops math and per-track no-overlap invariants are
sound. No Critical defects given the single-user, local, no-auth design.

The one issue that breaks a core flow end-to-end is **F1** (asset deletion
orphaning timeline references ‚Üí silent-black preview + hard-failing export with
an unactionable error) ‚Äî fix first. The cluster **F2/F3/F4** are real
correctness/durability gaps in the render + ingest lifecycle (hung SSE,
flip-back status, stuck rows) and should be addressed together. **F5/F6** are
robustness gaps (memory blow-up, lost-write autosave) reachable under large
media or fast editing. The remainder are low-impact hardening items.
